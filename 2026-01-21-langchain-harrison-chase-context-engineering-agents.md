## 核心观点

### 1. 长时程智能体 (Long-Horizon Agents) 的崛起与“上下文工程”
Harrison 指出，智能体（Agents）的概念从一开始就是让 LLM 在一个循环中运行并自主决定下一步操作。虽然这个想法早就存在（如 AutoGPT），但直到最近，随着模型能力的提升以及我们对“Harness”（驾驭系统/运行框架）理解的加深，这种长时程智能体才真正开始奏效。
其中最关键的概念是“上下文工程”（Context Engineering）。在传统的单次 LLM 应用中，开发者通过代码确切知道输入提示词（Prompt）和上下文是什么。但在智能体中，第 14 步的上下文取决于前 13 步的执行结果，这是动态且不可预测的。因此，如何管理、压缩和优化这个动态变化的上下文，成为了构建高效智能体的核心挑战。

### 2. Agent Framework (框架) vs. Agent Harness (驾驭系统)
Harrison 区分了 LangChain 早期的定位“框架”与现在的“Harness”。
- **Framework (框架)**：提供抽象层（如工具、向量数据库、内存的接口），但不预设具体实现方式，更加灵活。
- **Harness (驾驭系统)**：更加“电池包含”（batteries-included）且有明确的主张（opinionated）。例如，LangChain 的 Deep Agents 就是一种 Harness，它默认内置了规划工具（Planning）、上下文压缩机制（Compaction）以及文件系统交互能力。
Harness 的出现是因为我们发现某些特定的架构模式（如结合文件系统、特定的规划循环）是构建长时程智能体的最佳实践。

### 3. 文件系统是智能体的核心组件 ("File System Pilled")
Harrison 强烈认为（"completely file system pilled"），任何长时程智能体都需要访问某种形式的文件系统。
这不仅仅是为了写代码，更是为了管理上下文。例如，当上下文窗口快满时，可以将旧的消息或大型工具的输出结果转存到文件中，只保留引用。这样智能体在需要时可以去读取，而不会撑爆上下文。即使是不写代码的智能体，也可以使用基于数据库的“虚拟文件系统”来实现这种状态管理和上下文卸载。

### 4. 编程智能体 (Coding Agents) 是当前的最佳实践
目前长时程智能体最成功的应用领域是编程（Coding）。像 Claude Code、Factory 等产品展示了智能体如何通过长时间运行来解决复杂问题。
Harrison 认为，编程智能体的模式（使用终端、文件系统、循环调试）正在扩散到其他领域，如 AI SRE（站点可靠性工程）、深度研究（Deep Research）等。这些任务的共同点是：智能体产出一个“初稿”（First Draft），然后由人类进行审查和修改。这种“初稿生成”是目前长时程智能体的杀手级应用场景。

### 5. 软件开发 vs. 智能体开发：Trace (轨迹) 是新的真理来源
构建智能体与构建传统软件有本质区别。在传统软件中，逻辑写在代码里，看代码就知道程序会怎么跑。但在智能体中，大部分逻辑由模型在运行时决定，代码只是外壳。
因此，**Trace (执行轨迹)** 取代了代码成为“真理的来源”（Source of Truth）。当智能体出错时，开发者不能只看 GitHub 上的代码，必须查看 Trace 来了解模型在每一步的思考和上下文。Trace 也成为了团队协作、调试以及构建测试用例的核心载体。

### 6. 评估 (Eval) 的挑战与“LLM as a Judge”
智能体执行的任务通常需要人类判断（如“这份报告写得好不好”），因此传统的断言式测试（Assertions）不再适用。
解决方案引入“LLM as a Judge”（大模型作为裁判）。通过让模型模拟人类的判断标准来对 Trace 进行打分。为了确保准确性，需要进行“校准”（Alignment）：先由人类标注一部分数据，然后训练或优化 LLM 裁判，使其评分标准与人类一致。这是目前扩展智能体评估能力的有效路径。

### 7. 自我改进与“睡眠时间计算” (Sleep Time Compute)
Harrison 提到了一个令人兴奋的未来方向：智能体的自我改进。
目前的模式是：人类查看 Trace，发现问题，然后修改 Prompt 或代码。未来的模式是：智能体在夜间（"Sleep Time"）自动分析自己一天的 Trace，总结经验教训，并更新自己的指令文件（Instructions）或记忆库。这种递归式的自我改进（Recursive Self-Improvement）虽然目前还处于早期阶段（如 LangSmith Agent Builder 中的尝试），但潜力巨大。

### 8. 交互体验：异步 (Async) 与同步 (Sync) 的结合
长时程智能体的用户体验（UX）需要同时支持异步和同步模式。
- **异步**：因为任务可能运行很久（几小时甚至几天），用户不会一直盯着屏幕，需要像管理 Jira 工单或邮件一样异步管理这些任务。
- **同步**：当智能体完成“初稿”或遇到问题时，用户需要介入，通过聊天界面（Chat）进行实时的反馈和协作。
理想的界面应该是一个“收件箱”（Inbox）模式，既能概览后台运行的任务，又能随时切入具体的对话流进行干预，同时还能看到智能体操作的“状态”（如文件系统中的文件）。

## 核心问题 Q&A

### Q1: 什么是 "Agent Harness"？它与传统的 Agent Framework 有什么区别？
**A:** Harrison 解释说，LangChain 最初是一个 **Framework（框架）**，它提供了一套抽象层，让开发者可以方便地切换模型、连接工具和数据库，但它对“如何构建智能体”不持特定立场，非常灵活。
而 **Harness（驾驭系统/运行底座）**，如他们现在开发的 Deep Agents，则更加“固执”（opinionated）且功能完备。Harness 预设了最佳实践，例如：
1.  **内置规划工具**：默认赋予智能体规划能力。
2.  **上下文压缩 (Compaction)**：自动处理长对话中的上下文溢出问题。
3.  **文件系统集成**：默认提供文件系统读写能力，用于状态管理和上下文卸载。
简而言之，Framework 是工具箱，让你自己造车；Harness 是一辆已经造好的底盘，你只需要把数据和特定指令放进去就能跑。随着模型能力的收敛，Harness 这种封装了复杂上下文工程的标准化底座变得越来越重要。

### Q2: 为什么 Harrison 说自己被“文件系统药丸” (File System Pilled) 洗脑了？
**A:** Harrison 认为文件系统是长时程智能体不可或缺的组件，原因不仅在于写代码，更在于**上下文工程（Context Engineering）**。
长时程智能体在运行过程中会产生大量信息，很容易撑爆 LLM 的上下文窗口（Context Window）。文件系统提供了一种完美的“外挂内存”机制：
1.  **上下文卸载**：当工具返回海量数据（如几千行的日志）时，智能体不应该直接把这些数据塞进 Prompt，而是应该存入文件，只在 Prompt 里保留文件名。需要时再读取文件的特定部分。
2.  **状态持久化**：文件系统是保存“草稿”和“中间产物”的最佳场所，方便人类和智能体共同编辑。
即使是不涉及编程的任务，Harrison 也建议使用“虚拟文件系统”（Virtual File System，后端可以是数据库），因为这种“读写文件”的交互模式是目前管理复杂、动态上下文最高效的方式。

### Q3: 智能体开发与传统软件开发最大的区别是什么？
**A:** 最大的区别在于**逻辑的载体**和**确定性**。
- **传统软件**：逻辑完全由代码定义，是确定性的。要理解程序行为，你看代码就行。
- **智能体**：逻辑由“代码 + 模型 + 动态上下文”共同决定，是非确定性的黑盒。你无法仅通过阅读代码来预测智能体在第 14 步会做什么。
这就导致了开发流程的变革：
1.  **Trace (轨迹) 至上**：Trace 取代代码成为调试的核心。开发者需要像看慢动作回放一样去分析 Trace，看模型在每一步接收了什么上下文，输出了什么决策。
2.  **迭代式开发**：你无法在发布前完全“定义”智能体的行为，只能通过不断的运行、观察 Trace、调整 Prompt/工具来“塑造”它的行为。
3.  **测试的模糊性**：无法写简单的 `assert result == 5`，需要引入人类反馈或 LLM 裁判来评估结果的质量。

### Q4: 什么是“睡眠时间计算” (Sleep Time Compute)？它对智能体意味着什么？
**A:** “睡眠时间计算”是一个形象的比喻，指智能体在非工作时间（如夜间）利用计算资源进行自我反思和优化的过程。
Harrison 描述了这样一个愿景：
1.  白天，智能体执行任务，生成大量的 Trace（成功或失败的操作记录）。
2.  晚上，一个后台进程（或智能体本身）会扫描这些 Trace，分析哪里做得好、哪里犯了错。
3.  基于分析，智能体自动更新自己的 **System Prompt（系统提示词）** 或 **Few-Shot Examples（示例库）**。
这意味着智能体具有了“自我进化”的能力。开发者不需要手动去修每一个 Bug，智能体可以通过“做梦”（分析过去的经历）来从错误中学习。目前 LangSmith 的 Agent Builder 已经开始尝试这种机制，让智能体根据用户反馈自动修改自己的指令文件。

### Q5: 未来的智能体都会是“编程智能体” (Coding Agents) 吗？
**A:** 这是一个正在激烈讨论的话题。Harrison 倾向于认为，虽然不一定所有智能体都要写 Python 代码，但**通用智能体（General Purpose Agents）很可能会采用编程智能体的架构模式**。
目前的观察是，编程智能体（如 Claude Code）是表现最好的长时程智能体。它们使用终端、文件系统、循环试错的模式非常强大。
Harrison 指出，即使是处理非编程任务（如财务分析、资料搜集），赋予智能体“写脚本”的能力也是极具价值的，因为代码是处理数据和控制计算机最通用的语言。
因此，未来的通用智能体可能看起来很像现在的编程智能体：它们都有一个“沙箱”，都能读写文件，都能执行某种形式的脚本语言来完成任务。

所以就像我们上次聊的，Prompt 需要针对不同的模型进行调整，Harness 也需要针对不同的模型家族做微调。
但它们也有相似之处。它们都在某种意义上使用文件系统。
所以我其实不知道这个问题的答案。这很有趣。我们看到现在每个做编程公司的人基本上都在构建自己的 Harness。
而且现在有各种排行榜，如果你去看 Terminal Bench 2，这可能是目前比较流行的编程基准测试之一。你可以看到他们列出了 Agent Harness 和 Model，你可以看到性能的差异，而 Claude Code 并不在榜首。
所以是有差异的。但这并不一定意味着模型实验室（Model Labs）更擅长做这个。这只是意味着你需要理解模型是如何工作的，那些研究如何让 Harness 配合模型运作的人可以在那里获得性能提升。

**主持人**: 你认为是什么让 Harness 运作良好？排行榜顶端的那些人做对了什么？

**Harrison**: 我觉得一部分绝对是理解模型是基于什么工具训练的。比如 OpenAI 在 Bash 上训练得很重，Anthropic 有显式的文件编辑工具，所以顺应这些是关键。
压缩（Compaction）也变得越来越重要。特别是当你开始做长时程任务时，你会填满上下文窗口，所以怎么处理这个问题是个大问题，有很多策略。我会说这也是 Harness 的一部分。
所有的 Harness 也是 Skills（技能）、MCP（模型上下文协议）和 Sub-agents（子智能体）开始发挥作用的地方。你可以以不同的方式使用它们。我不认为模型里已经训练了很多关于 Skills 或 Sub-agents 的知识，那些还很新。
我们在我们的 Harness 中看到的一件事是，当你有一个子智能体时，主模型需要与它通信，需要给它适当的信息，需要让子智能体知道它需要把最终结果返回出来。
我们看到过一些失败模式，子智能体做了一堆工作，然后基本上说“看我上面的工作”，然后把这个传回给主智能体。但主智能体看不到上面的工作（因为上下文隔离），它就会问“你在说什么？”。
所以，编写 Prompt 让这些部分协同工作是很大的一部分。Skills、Sub-agents、MCP，所有这些 Harness 里都有成百上千行的 Prompt 来让它们工作得好或不好。

**主持人**: 我能问一个关于演变的问题吗？你一直处于人们如何让模型在现实世界中工作的最前沿。如果我们用简化的视角看过去五年的重大拐点：ChatGPT 出来时是预训练（Pre-training）的拐点；o1 出来时是推理（Reasoning）的拐点；最近 Claude Code 和 Opus 4.5 似乎是长时程智能体的第三个拐点。

**Harrison**: 我觉得大概有三个时代。
早期，也就是 LangChain 刚开始的时候，还是原始的“文本进文本出”，甚至不是对话模型。它们没有工具调用，没有内容块，没有任何推理能力。它们真的非常基础。当时人们做的主要是单一 Prompt 或简单的链（Chains）。
然后，很多模型实验室开始在模型中训练工具调用（Tool Calling），他们试图让模型擅长思考和规划。虽然当时还不够好，不如今天这么好，但已经足够用来决定做什么了。这就是自定义认知架构（Custom Cognitive Architectures）开始发挥作用的地方。你会明确地问它“我现在该做什么？”，然后进入这个分支，或者那个循环。开始有一些循环，但更多的是脚手架。
然后有一个拐点，我不知道确切是什么时候。大概是今年六七月份，我们看到 Claude Code 起飞，Deep Research 起飞，Manus 起飞。它们底层使用的是相同的架构：就是 LLM 在循环中运行。但巧妙的是——很多 Harness 其实就是上下文工程（Context Engineering）。围绕压缩的一切是上下文工程，子智能体是上下文工程，Skills 是上下文工程。
我们看到它们使用相同的核心算法，但在上下文工程上做了改进。我们当时想，哦，这很有趣，这和以前很不一样。这也是我们开始做 Deep Agents 的时候。
对于编程社区的很多人来说，Opus 4.5 可能是他们真正感受到这一点的时候。也可能只是恰逢寒假，大家回家开始用 Claude Code，才发现它有多好。
但在 11 月、12 月左右，我感觉到一个巨大的“氛围转变”（Vibe Shift）。人们开始觉得，是的，你把难题扔给这些东西，你会得到长时程智能体。
所以我不知道是 2025 年初还是 2025 年末，但在某个时刻，模型变得足够好了，那就是我们从 Scaffolds（脚手架）转向 Harnesses（驾驭系统）的时候。

**主持人**: 这条弧线上的下一步是什么？

**Harrison**: 我真希望我能告诉你。我的意思是，我认为这种让 LLM 在循环中运行、让它自己编排、让它真正选择把什么拉入上下文的算法，它是如此简单且通用。这就是智能体一直以来的核心理念，我们终于做到了。
如果你看一些手动的脚手架，也许其中一些会消失。比如压缩（Compaction）现在还是很手动的，Harness 的作者决定怎么做。Anthropic 有一些有趣的东西，让模型决定何时压缩。我们还没看到很多人用这个，也许那是下一步。
我们非常感兴趣的另一部分是记忆（Memory）。如果你在这个背景下思考记忆，那也是上下文工程，对吧？它是更长时间跨度上的上下文工程，是一组稍微不同的上下文，但仍然是提供给 LLM 的。
我认为核心算法非常简单：在循环中运行 LLM。我们终于到了它能工作的地步。所以会有很多围绕它的上下文工程技巧。也许其中一些是将上下文工程本身交给 LLM，比如 Anthropic 的做法。也许只是引入新类型的上下文。
模型可能会变得更好，它们可能会越来越擅长这些长时程任务。
我脑子里的一个大问题是，我们看到的很多 Harness 都是非常针对编程的（Coding Specific）。那是我们最早看到长时程智能体起飞的地方。即使对于非编程任务，你也可以争辩说写代码是非常有用的，可以是通用的。

**主持人**: 我正想问你，编程智能体是一个子类，还是说编程智能体就是智能体？意思是，智能体的工作是想办法让计算机做有用的事，而代码是让计算机做有用事的很好的方式。

**Harrison**: 我不知道。这是最大的问题之一。我非常非常强烈地相信，如果你现在在构建一个长时程智能体，你需要给它访问文件系统的权限。
文件系统在上下文管理方面能做的事情太多了。当我们谈论压缩时，一种策略是总结，但把所有消息都放在文件系统中，这样如果它需要查阅，它可以去查。
另一种策略是，当你有巨大的工具调用结果时，不要把它全部传给模型。把它放在文件系统中，让模型去查阅。
现在，你可以不用真正的文件系统，不用让它写代码就能做到这一点。我们有一个“虚拟文件系统”的概念，后端是 Postgres 之类的，更具扩展性。
但显然有些事情你可以用代码做，而不能用虚拟文件系统做。你不能在虚拟文件系统里运行代码。写脚本对此非常有用。
我认为编程智能体可以是通用的，但我不知道这是否意味着今天的编程智能体是通用的。因为我认为今天的很多编程智能体对编程任务进行了过度优化。
所以我觉得通用的智能体可能是一个编程智能体，但我不知道反过来是否成立。
我们也在思考这个问题：是不是所有的智能体都是编程智能体？这是我们现在思考的最大的事情之一。

**主持人**: 也许我们可以转而谈谈构建长时程智能体与构建软件有什么不同？你能描述一下 1.0 代码开发的软件开发栈，以及现在有什么不同吗？你在 X 上有一篇很好的文章谈这个。也许总结一下重点。

**Harrison**: 我一直在思考这个问题，因为我们喜欢说构建智能体不同于构建软件，而且很多人也同意。但到底哪里不同？说它不同很容易也很懒惰，但实际上有什么不同？
这些可能听起来很明显，但希望它们没有争议。
当你构建软件时，所有的逻辑都在软件的代码里，你可以看到它。
当你构建智能体时，应用程序如何工作的逻辑并不全在代码里。很大一部分来自模型。
这意味着你不能只看代码就知道智能体在特定场景下会做什么。你必须实际运行它。
我认为这是最大的区别。我们引入了这些非确定性的系统，它是一个黑盒，它存在于外部。
那么这到底意味着什么？
我认为其中一件事是，为了知道应用程序实际上在做什么，你不能看代码。你必须看它在现实生活中实际做了什么。
这就是为什么我们做的最受欢迎的事情之一是 LangSmith。其中的核心部分是 Tracing（追踪）。为什么 Trace 如此受欢迎？因为它们确切地告诉你智能体内部每一步发生了什么。
这与软件的 Trace 不同。在软件中，你通常只在生产环境出问题时才打开它。但在智能体中，人们从一开始就使用 Trace 来了解底层发生了什么。
而且这在智能体中比在单一 LLM 应用中影响更大。因为在单一 LLM 应用中，输入和上下文是由代码决定的。在智能体中，它们在循环运行，你不知道第 14 步的上下文是什么，因为前 13 步可能引入了任意东西。
Trace 告诉你上下文里有什么。这太重要了。
这意味着，软件的真理来源（Source of Truth）在代码里。而对于智能体，真理来源现在是代码和 Trace 的结合。
这也意味着 Trace 成了你开始思考测试的地方。你不能只做离线单元测试，你可能想用 Trace 来构建测试用例。在线测试（Online Testing）在智能体中比在软件中更重要，因为行为直到真正使用真实世界输入时才会涌现。
我们看到 Trace 正在成为团队协作的节点。如果出了问题，不是说“去 GitHub 看代码”，而是“去看 Trace”。我们在开源社区也看到，当人们说“Deep Agents 跑偏了，发生了什么？”我们的反应是“发给我们一个 LangSmith Trace”。如果是以前，我们会说“给我看代码”。
另一件事，我也还在思考，我认为构建智能体更具迭代性（Iterative）。
我们以前也这么说，我会翻白眼，因为构建软件也是迭代的，对吧？你发布，获得反馈，不断迭代。
但我认为区别在于，在软件中，你是基于你想让软件做什么来迭代。你有想法，发布，获得反馈，也许按钮位置不对，也许用户想做 X 而不是 Y，但在发布前你知道软件是做什么的。
对于智能体，在发布前你并不知道智能体实际上会做什么。你有个想法，但你真的不知道。
所以为了让它准确、正确、通过概念上的单元测试，需要更多的迭代。
基于此，这也是我认为记忆（Memory）非常重要的原因。因为记忆就是从这些交互中学习。如果现在的过程更加迭代，作为开发者我必须修改 System Prompt 的次数远多于修改代码的次数。
如果系统可以自我学习，那就能减少开发者的迭代工作量。这就是记忆的作用。

**Harrison**: 连老家伙们都懂了。

**主持人**: 好的。看起来 Trace 是你认为在这个新世界里的核心工件（Artifact）。还有其他的核心工件吗？特别是我想问问 Eval（评估）。

**Harrison**: 是的，也许“工件”这个词不太对，应该是“组件”。
我认为构建软件和构建智能体的另一个区别是，评估软件你可以依赖测试和断言（Assertions）。

这是一个很好的心理框架：这是你的工作区（Workspace）。它可以是 Google Drive，可以是 Notion 页面。
你启动它，异步管理，然后进入同步模式与它聊天，同时也查看这个状态。这就是我目前看到的。

**主持人**: 这就像你的 Agent Inbox（智能体收件箱）的想法。为了启用同步模式，你的智能体需要一种联系你的方式。

**Harrison**: 完全正确。我们大约一年前推出了 Agent Inbox，当时的想法是“环境智能体”（Ambient Agents）在后台运行并 ping 你。
第一个版本没有同步模式，它 ping 你，你回复，然后你只能等它再次 ping 你。但这很烦，因为我想进入同步模式，我想和它对话。

所以我认为纯异步模式目前行不通。也许未来它们变得足够好，不需要你纠正时可以。但至少现在，我们看到人们在异步和同步之间来回切换。

编程，我也许没那么被洗脑，但我大概 90% 确信。对于长尾用例，编程很棒，真的没有替代品。
浏览器使用，目前的模型还不够好。你也许可以给编程智能体一个 CLI 来做浏览器使用，可能有某种近似。
